# Implementation Spec for Codex

- **Author:** L. Saetta
- **Version:** 0.1.0
- **Last modified:** 2026-05-02

## 1. Document Purpose

This document guides Codex-assisted development of the tool for automated deployment of containerized AI components on OCI Enterprise AI.

The main design document describes the architecture, rationale, and functional flow. This implementation spec translates that design into incremental tasks, acceptance criteria, repository structure, test commands, and implementation constraints.

Reference document:

```text
./design_deploy_oci_enterprise_ai.md
```

## 2. Version 1 Goal

Version 1 must produce a Python command-line tool that reads a declarative YAML file and orchestrates deployment through Docker and the OCI CLI.

The target flow is:

```text
YAML -> Python -> generated JSON -> OCI CLI -> OCI Enterprise AI
```

Version 1 must support:

- reading and validating the YAML file
- reading a single-deployment legacy YAML file
- reading an Enterprise Solution YAML file with one or more deployments
- enforcing one region and one compartment for the whole Enterprise Solution
- executing Enterprise Solution deployments serially
- generating intermediate JSON files for OCI CLI
- `dry run` mode
- building the Docker image
- tagging the image for OCIR
- pushing the image to OCIR
- creating or reusing the Hosted Application
- creating the Hosted Deployment
- waiting for the final state, where supported by the OCI CLI
- readable error handling
- automated tests for parts that do not depend on real OCI resources

For Enterprise Solution YAML files, every deployment must have its own dedicated
Hosted Application and Hosted Deployment. The relationship is one-to-one:

```text
deployment[0] -> Hosted Application[0] -> Hosted Deployment[0]
deployment[1] -> Hosted Application[1] -> Hosted Deployment[1]
```

The implementation must not create multiple Hosted Deployments under one shared
Hosted Application in this version.

## 3. Non-Goals for Version 1

Version 1 must not implement:

- direct use of the OCI Python SDK
- web UI
- mandatory interactive menu
- automatic creation or modification of IAM policies
- automatic creation of secrets in OCI Vault
- automatic rollback based on application health checks
- clear-text storage of secrets
- automatic inference of missing IDCS auth configuration
- parallel deployment execution
- shared Hosted Applications for multiple deployment items

IAM policies and working OCI permissions are external prerequisites. They must be prepared by OCI admins before using the tool.

## 4. Implementation Principles

- The YAML file is the human-readable declarative source.
- Generated JSON files are intermediate technical artifacts.
- The OCI CLI is the operational engine in version 1.
- Python orchestrates the flow and calls Docker and the OCI CLI through `subprocess`.
- Every command must be usable in CI/CD.
- Deployment command plans may be exported to an executable POSIX shell script
  for Linux and macOS review or handoff.
- Every destructive or expensive operation must be visible in `dry run`.
- The tool must not print secrets.
- Errors must be readable and actionable.
- Logic must be split into testable modules.

## 6. Command-Line Interface

Base command:

```bash
python oci_ai_deploy.py --config examples/agent_dev.yaml validate
```

Required commands:

```bash
python oci_ai_deploy.py --config examples/agent_dev.yaml validate
python oci_ai_deploy.py --config examples/agent_dev.yaml render
python oci_ai_deploy.py --config examples/agent_dev.yaml build
python oci_ai_deploy.py --config examples/agent_dev.yaml push
python oci_ai_deploy.py --config examples/agent_dev.yaml create-application
python oci_ai_deploy.py --config examples/agent_dev.yaml create-deployment
python oci_ai_deploy.py --config examples/agent_dev.yaml deploy
python oci_ai_deploy.py --config examples/agent_dev.yaml rollback --to-tag abc1234
```

When the YAML file uses the Enterprise Solution shape, `render`, `build`,
`push`, `create-application`, and `deploy` apply to all deployment items in the
declared order. `deploy` must stop immediately on the first deployment failure
and report the failed deployment name and phase.

Required global parameters:

```text
--config <path>
--dry-run
--non-interactive
--output-dir <path>
--script-file <path>
--verbose
```

Expected behavior of `--dry-run`:

- does not run `docker build`
- does not run `docker push`
- does not call OCI CLI to create or update resources
- may call non-invasive local commands, if needed, to validate prerequisites
- assumes Docker login to the target OCIR registry has already been completed;
  `validate` checks the local Docker configuration and reports a missing login
- prints the commands that would be executed
- generates intermediate JSON files in the output directory, if requested
- when `--script-file` is provided with `deploy`, writes an executable `.sh`
  script containing the generated deploy commands and JSON file references
- masks secrets and sensitive values
- returns exit code `0` if configuration is valid and the plan can be built

Example:

```bash
python oci_ai_deploy.py --config examples/agent_dev.yaml deploy --dry-run
```

## 7. YAML Format Supported in Version 1

Version 1 must support the legacy single-deployment structure:

```yaml
application:
  name: my-agent-app
  compartment_id: ocid1.compartment.oc1..example
  region: eu-frankfurt-1
  region_key: fra

container:
  context: .
  dockerfile: Dockerfile
  image_repository: ai-agents/my-agent
  tag_strategy: git_sha
  ocir_namespace: auto

hosted_application:
  display_name: my-agent-app
  description: Agent application deployed through OCI Enterprise AI
  create_if_missing: true
  update_if_exists: false

  scaling:
    min_instances: 1
    max_instances: 2
    metric: cpu

  networking:
    mode: public

  security:
    auth_type: IDCS_AUTH_CONFIG
    issuer_url: https://issuer.example.com
    audience: my-agent-api
    scopes:
      - my-agent-api/.default

  environment:
    variables:
      LOG_LEVEL: INFO
      OCI_REGION: eu-frankfurt-1
      AGENT_MODE: production
      MCP_SERVER_PORT: "8080"

    secrets:
      API_KEY:
        source: vault
        secret_ocid: ocid1.vaultsecret.oc1..example

hosted_deployment:
  display_name: my-agent-deployment
  create_new_version: true
  activate: true
  wait_for_state: SUCCEEDED
```

`compartment_name` is also accepted as an alternative to `compartment_id`.
Exactly one of the two fields must be present. When `compartment_name` is used,
the command-line tool and Web UI resolve it through OCI IAM compartment lookup
before validation, rendering, or deployment proceeds.

Version 1 must also support this Enterprise Solution structure:

```yaml
enterprise_solution:
  name: my-ai-solution
  compartment_id: ocid1.compartment.oc1..example
  region: eu-frankfurt-1
  region_key: fra

deployments:
  - name: agent-api
    container:
      context: ./agent-api
      dockerfile: Dockerfile
      image_repository: ai-agents/agent-api
      tag_strategy: git_sha
      ocir_namespace: auto

    hosted_application:
      display_name: agent-api-app
      description: Agent API deployed through OCI Enterprise AI
      create_if_missing: true
      update_if_exists: false
      networking:
        mode: public
      security:
        auth_type: IDCS_AUTH_CONFIG
        issuer_url: https://issuer.example.com
        audience: agent-api
        scopes:
          - agent-api/.default
      environment:
        variables:
          LOG_LEVEL: INFO

    hosted_deployment:
      display_name: agent-api-deployment
      create_new_version: true
      activate: true
      wait_for_state: SUCCEEDED

  - name: mcp-server
    container:
      context: ./mcp-server
      dockerfile: Dockerfile
      image_repository: ai-agents/mcp-server
      tag_strategy: git_sha
      ocir_namespace: auto

    hosted_application:
      display_name: mcp-server-app
      create_if_missing: true
      update_if_exists: false
      networking:
        mode: public
      security:
        auth_type: NO_AUTH
      environment:
        variables:
          LOG_LEVEL: INFO

    hosted_deployment:
      display_name: mcp-server-deployment
      create_new_version: true
      activate: true
      wait_for_state: SUCCEEDED
```

Enterprise Solution constraints:

- `enterprise_solution.region` applies to every deployment
- `enterprise_solution.region_key` applies to every deployment
- `enterprise_solution.compartment_id` applies to every deployment
- every item in `deployments` must have a unique `name`
- every item in `deployments` must have a unique Hosted Application display name
- generated artifacts for multiple deployments must be written under
  deployment-specific output directories

## 8. Secrets Handling

Secrets must not be written in clear text in the YAML file.

For local development, reading secrets from environment variables or from a `.env` file excluded from version control is acceptable. For shared environments or CI/CD, OCI Vault or the pipeline secret manager is preferable.

Version 1 must:

- support references to secrets, not clear-text values
- support `source: vault`
- support `source: local_env` for local development
- avoid printing sensitive values in logs
- mask fields with suspicious names, for example `secret`, `token`, `password`, `api_key`, `client_secret`
- fail validation if it finds obvious hardcoded secrets in the YAML

Acceptable local example:

```yaml
secrets:
  API_KEY:
    source: local_env
    env_name: MY_API_KEY
```

Example to avoid:

```yaml
secrets:
  API_KEY: my-secret-value
```

## 9. Runtime State and Generated Output

The YAML file describes the desired state. It must not be modified by the script to save OCIDs, endpoints, or work requests.

The script may generate separate state files, for example:

```text
generated/state/my-agent-app-dev.json
```

Indicative content:

```json
{
  "application_id": "ocid1...",
  "deployment_id": "ocid1...",
  "image_uri": "fra.ocir.io/example/ai-agents/my-agent:abc1234",
  "endpoint": "https://example.endpoint",
  "created_at": "2026-04-29T10:00:00Z",
  "oci_work_request_id": "ocid1..."
}
```

The final deployment report should be written to:

```text
generated/deploy-report.json
```

## 10. Incremental Tasks for Codex

### Task 1, create CLI skeleton

Implement the project structure, the `oci_ai_deploy.py` file, and the `cli.py` module.

Acceptance criteria:

- `python oci_ai_deploy.py --help` works
- all required commands are visible in help
- `--config` is accepted as a global parameter
- `--dry-run` is accepted as a global parameter
- each command invokes a dedicated function, even if initially stubbed
- CLI tests verify help and argument parsing

### Task 2, YAML configuration parsing

Implement `config.py`.

Acceptance criteria:

- the tool reads YAML from a path
- missing-file errors are readable
- YAML errors are readable
- configuration is represented with dataclasses or typed structures
- tests cover valid YAML and malformed YAML

### Task 3, configuration validation

Implement `validation.py` and, if useful, `schemas/oci_ai_deploy.schema.json`.

Acceptance criteria:

- `validate` checks required fields
- `validate` accepts only `auth_type: IDCS_AUTH_CONFIG` or `auth_type: NO_AUTH`
- `validate` checks consistency between `auth_type: IDCS_AUTH_CONFIG` and required auth fields
- `validate` checks Dockerfile presence
- `validate` checks minimal compartment OCID syntax
- `validate` checks that region and region key are set
- `validate` blocks obvious hardcoded secrets
- `validate` returns a non-zero exit code on error
- tests cover valid and invalid cases

### Task 4, tag and OCIR image URI calculation

Implement `ocir.py`.

Acceptance criteria:

- supports `tag_strategy: git_sha`
- supports `tag_strategy: timestamp`
- supports explicit tag, if provided by the YAML
- builds image URI in the format `<region_key>.ocir.io/<namespace>/<image_repository>:<tag>`
- if `ocir_namespace: auto`, prepares the OCI CLI command required to retrieve it
- in `dry run`, if the namespace is not known, clearly shows that it will be retrieved at runtime
- tests cover URI construction and tag strategies

### Task 5, JSON rendering for OCI CLI

Implement `renderer.py`.

Acceptance criteria:

- `render` generates JSON for Hosted Application
- `render` generates JSON for Hosted Deployment
- files are written to `generated/` or to `--output-dir`
- JSON files do not contain clear-text secrets
- `auth_type: NO_AUTH` renders an explicit inbound auth config using
  `inboundAuthConfigType: NO_AUTH_CONFIG`, because the public OCI CLI/SDK model
  may lag this Limited Availability feature
- the Hosted Application versus Hosted Deployment mapping follows the design table
- fields still marked `TBD` are isolated and commented in code
- tests compare JSON output with expected fixtures

### Task 6, safe subprocess wrapper

Implement `oci_cli.py` and a common utility for command execution.

Acceptance criteria:

- commands are built as argument lists, not shell strings
- stdout and stderr are captured
- subprocess errors are converted into readable messages
- secrets are masked in logs
- in `dry run`, the command is printed but not executed
- tests use `subprocess.run` mocks

### Task 7, Docker operations

Implement `docker_ops.py`.

Acceptance criteria:

- `build` builds the `docker build` command
- `push` ensures the OCIR container repository exists before pushing, creating
  `<image_repository>` in the configured compartment when it is missing
- `push` builds the `docker push` command
- the OCIR tag is applied correctly
- in `dry run`, Docker is not invoked
- Docker errors are readable
- tests use subprocess mocks

### Task 8, Hosted Application

Implement Hosted Application creation or reuse logic in `deploy.py`.

Acceptance criteria:

- the tool builds a list or lookup command for the Hosted Application
- if it exists, it retrieves the application OCID
- if it does not exist and `create_if_missing` is true, it creates the Hosted Application
- if it does not exist and `create_if_missing` is false, it fails with a clear message
- `update_if_exists` is respected
- in `dry run`, it shows the plan without calling OCI
- tests cover the main branches using mocks

### Task 9, Hosted Deployment

Implement Hosted Deployment creation logic in `deploy.py`.

Acceptance criteria:

- the tool builds the command for Hosted Deployment with single Docker artifact, if confirmed by the target OCI CLI
- uses Hosted Application OCID
- uses calculated image URI
- supports `activate`
- supports `wait_for_state`, if available in the CLI
- saves deployment OCID if returned by the CLI
- in `dry run`, it shows the plan without calling OCI
- tests cover command construction and output parsing

### Task 10, end-to-end deploy command

Compose validate, render, build, push, create application, and create deployment.

Acceptance criteria:

- `deploy` runs phases in the correct order
- `deploy --dry-run` shows all phases without remote side effects
- `deploy --script-file generated/deploy.sh` writes an executable shell script
  after JSON artifacts and deploy commands are generated
- an error stops the flow with a clear message
- the final report is produced when possible
- tests verify phase ordering through mocks

### Task 11, basic rollback

Implement rollback based on immutable tag.

Acceptance criteria:

- `rollback --to-tag <tag>` calculates the image URI with the requested tag
- creates a new Hosted Deployment that points to the given tag
- does not modify the YAML file
- in `dry run`, shows the deployment that would be created
- tests cover URI calculation and command construction

### Task 12, README and operational documentation

Update README.

Acceptance criteria:

- README explains prerequisites
- README explains Python dependency installation
- README explains validation
- README explains dry run
- README explains deploy
- README explains rollback
- README clarifies that OCI policies must be prepared by OCI admins
- README includes examples but no real secrets

## 11. Definition of Done

Version 1 is complete when:

- all mandatory tasks are implemented
- `python -m pytest tests` passes
- `python oci_ai_deploy.py --config examples/agent_dev.yaml validate` works
- `python oci_ai_deploy.py --config examples/agent_dev.yaml render` generates valid JSON
- `python oci_ai_deploy.py --config examples/agent_dev.yaml deploy --dry-run` produces no remote side effects
- no secret is printed in logs or generated JSON
- main errors are readable
- the README allows a new developer to use the tool locally
- remaining `TBD` points are explicit and documented

## 13. Code Conventions

- Use Python 3.11 or higher.
- Use `argparse` for the CLI in version 1, unless a different explicit decision is made.
- Use `dataclasses` or simple typed models for configuration.
- Avoid unnecessary dependencies.
- Use `PyYAML` to read YAML.
- Use `pytest` for tests.
- Do not use `shell=True` in `subprocess.run`.
- Centralize logging and secret masking.
- Separate command construction from command execution to make testing easier.
- Keep modules small and focused on a single responsibility.

## 14. Error Handling

Errors must indicate:

- failed phase
- involved command, with secrets masked
- exit code, if available
- concise stderr
- operational suggestion, when possible

Example:

```text
Error in create-deployment phase.
The OCI CLI command returned exit code 1.
Verify that the Hosted Application exists, that the image URI is accessible by the service, and that OCI policies are correct.
```

## 15. Logging

Minimum levels:

- normal, concise output for human use
- verbose, diagnostic output with commands and generated files
- dry run, operational plan without side effects

Logging must never print secrets.

Fields to always mask if present:

```text
password
secret
token
api_key
apikey
client_secret
authorization
```

## 16. OCI Policies and External Prerequisites

The tool assumes that OCI policies are already correct.

Prerequisites to document:

- user or principal authorized to use OCI CLI
- permissions to push to OCIR
- permissions to create or manage Hosted Applications
- permissions to create or manage Hosted Deployments
- permissions to read secrets from Vault, if used
- permissions allowing the service to read the image from OCIR
- optional permissions for Logging, Networking, and diagnostics

The tool can validate some technical prerequisites, but it must not try to create or modify IAM policies.

## 17. TBD Points to Verify Before Real Deployment

Before using the tool in production, verify:

- exact name and availability of the OCI CLI commands for Hosted Application
- exact name and availability of the OCI CLI commands for Hosted Deployment
- JSON format required by `create-hosted-deployment-single-docker-artifact`
- correct location of environment variables between Hosted Application and Hosted Deployment
- correct location of scaling between Hosted Application and Hosted Deployment
- correct location of IDCS auth between Hosted Application and Hosted Deployment
- actual endpoint model returned by OCI
- actual behavior of `wait_for_state`
- actual work-request output

## 18. Recommended Way to Use Codex

Do not ask Codex to implement everything in a single step.

Proceed incrementally:

- give Codex one task at a time
- ask for tests together with code
- run tests after each task
- make small commits
- update this spec when real OCI CLI details emerge
- keep `TBD` points explicit

Recommended prompt for the first task:

```text
Use implementation_spec_codex.md as the operational specification.
Implement only Task 1, create CLI skeleton.
Add pytest tests for help and argument parsing.
Do not implement Docker or OCI CLI calls yet.
```

Recommended prompt for subsequent tasks:

```text
Continue from the implementation spec.
Implement only Task <number>.
Keep subsequent tasks unchanged.
Add or update the relevant tests.
Mentally verify the change against the task acceptance criteria.
```

## 19. Review Criteria for Generated Code

Every PR or commit generated by Codex should be checked against these criteria:

- implements only the requested scope
- adds relevant tests
- does not introduce secrets or real data
- does not use `shell=True`
- does not make `dry run` impossible
- does not confuse desired configuration with runtime state
- does not automatically modify OCI policies
- keeps errors readable
- preserves separation between modules

## 20. Final Note

This implementation spec is intentionally more operational than the design document. It must evolve together with the code, especially when the real OCI CLI details and the JSON format required by Hosted Application and Hosted Deployment resources are verified.
